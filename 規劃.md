# FlexQueue 規劃文件

---

## 1. 專案定位

- **目標**：提供一套可在 Joomla 或其他 PHP 專案重複使用的任務佇列 library。
- **用途**：讓 component / plugin / CLI 工具可用一致介面 dispatch 與處理工作，視需求選擇不同後端 driver。
- **設計原則**：
  - library 本身僅提供核心程式碼與介面，不內含 DI 或框架耦合邏輯。
  - 所有外部依賴透過介面抽象，由 consumer 或官方 plugin 透過容器注入。
  - 保持最小核心，功能以組件方式擴充。

---

## 2. 預期使用情境

- 中大型 Joomla 專案需要統一的背景工作框架。
- 第三方 extension 想整合 Redis/RabbitMQ 等訊息佇列。
- DevOps 需將工作轉交給外部 worker，但仍維持 Joomla 內寫的 Job handler。

---

## 3. 架構概覽

- **Library 階層**：

```
FlexQueue (Library)
│
├── Contracts/
│   ├── JobInterface
│   └── QueueDriverInterface
├── Drivers/
│   ├── DatabaseQueueDriver
│   ├── RedisQueueDriver
│   └── RabbitMQQueueDriver
├── Support/
│   ├── QueueManager
│   └── DriverFactory
└── Worker/
    └── QueueWorker (選用 CLI)
```

- **Bridge Plugin**：提供官方 Joomla system plugin（或其他框架 adapter），讀取部署設定後，透過容器的 `set` / `share` 閉包把 library 綁定進應用層。
- **Core Contracts**：定義工作、driver 與設定介面。
- **Support Layer**：負責工廠與輔助工具，純 PHP，不依賴容器。
- **Drivers**：具備最少相依的實作模板，真正的連線細節透過注入的工具或外部依賴完成。

---

## 4. 設定策略

### 4.1 設定策略

- library 僅需得知 driver 名稱與對應設定。
- 由 consumer（Joomla plugin）直接以 params 組合設定陣列，不再引入額外 repository 介面。

### 4.2 消費者責任

- consumer 負責提供設定來源：
  - `Registry` / `#__extensions.params`
  - `.env` 或自訂設定檔
  - runtime 計算出的部署參數
- 應確保容器能產生 `QueueDriverInterface` 所需資訊，通常為 driver 名稱與相關配置陣列。

---

## 5. DI 與橋接層

### 5.1 Library 立場

- library 不再內含 `ServiceProvider`，也不主動存取任何容器。
- 只提供 `DriverFactory`、`QueueManager` 等純 PHP 類別，讓外部自行組合。

### 5.2 官方橋接 plugin（Joomla 範例）

- System plugin 於 bootstrap 期讀取 plugin params，透過 `Factory::getContainer()` 取得 Joomla 容器。
- 利用 `Container::set()` / `share()` 閉包直接註冊 FlexQueue 所需依賴：
  - 在閉包內解析 params，產生 driver 需要的設定陣列。
  - 呼叫 `DriverFactory::make()` 建立 `QueueDriverInterface` 實例。
  - 最後註冊 `QueueManager`，供其他 extension 注入使用。

```php
use FlexQueue\Contracts\QueueDriverInterface;
use FlexQueue\Support\DriverFactory;
use FlexQueue\Support\QueueManager;
use Joomla\CMS\Factory;
use Joomla\DI\Container;

public function onAfterInitialise(): void
{
    $container = Factory::getContainer();
    $params = $this->params;

    $container->share(QueueDriverInterface::class, function (Container $c) use ($params): QueueDriverInterface {
        $factory = new DriverFactory();

        $config = [
            'flexqueue' => array_merge(
                ['driver' => $params->get('driver', 'database')],
                (array) json_decode((string) $params->get('config_json', '{}'), true)
            ),
        ];

        return $factory->make($config['flexqueue']['driver'], $config);
    });

    $container->share(QueueManager::class, fn (Container $c) => new QueueManager(
        $c->get(QueueDriverInterface::class)
    ));
}
```

### 5.3 Consumer 註冊策略

- 在 Joomla 以外的環境，consumer 可直接於容器閉包內組合設定陣列。
- 若有既有 config 服務（如 Symfony、Laravel），可撰寫 adapter 將設定轉成陣列後再傳給 `DriverFactory::make()`。
- 核心精神：library 不持有容器，所有 DI 相關邏輯都放在外部 bridge。

### 5.4 DriverFactory

- 根據 driver 字串決定建立哪個 driver 實例。
- 對於需要外部連線的 driver，再考慮注入連線依賴或使用協力程式類別（例如 `RedisConnectionFactory`）。

---

## 6. Driver 設計

- **DatabaseQueueDriver**：
  - 僅依賴一個 `DatabaseAdapterInterface`（自行定義），consumer 必須提供符合介面的資料庫連線。
  - queue table schema 維持簡單，提供 migration SQL 參考。
- **RedisQueueDriver** / **RabbitMQQueueDriver**：
  - 接受連線工廠或現成 client 實例，避免在庫內建構具體 client。
  - 提供基本的 `push` / `pop`，延遲與重試策略另行擴充。

---

## 7. 任務與 Worker

- Job handler 只負責執行業務邏輯，必要依賴透過建構子或 DI 注入。
- Worker 可作為 CLI 工具，示範如何：
  - 從 container 取出 `QueueDriverInterface`
  - 迴圈 pop message
  - 反射/Factory 建立 Job handler 並執行

---

## 8. 整合情境

### 8.1 Joomla Component

- 在 `administrator/components/com_xxx/services/provider.php` 內註冊 FlexQueue provider。
- 設定可儲存在 component params，後端表單讓使用者編輯。

### 8.2 Joomla Plugin

- plugin 啟動時載入設定（例如 system plugin `onAfterInitialise`）。
- 透過 container 注入 driver，提供事件 hook 將任務送進 queue。

### 8.3 獨立 CLI / Laravel 等其他框架

- 只要建立 `Container` 或對應 DI 容器，註冊 provider 即可。
- 若框架已有 config 服務，可寫一個 adapter 將設定轉成陣列。
